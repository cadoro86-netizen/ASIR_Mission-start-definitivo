# ASIR_Mission-start-definitivo
Acabas de ser contratado como Desarrollador Junior en NextGen Web Solutions. Tu primera misión no es escribir código desde cero, sino preparar tu "estación de combate" digital y resolver un problema de rendimiento que está afectando a un cliente importante.  +1
MILESTONE 1 – Estación de Combate Profesional
🔹 Issue 1.1 – Configuración base de VS Code

Título: Configuración profesional del entorno de desarrollo en VS Code

Descripción:
Como desarrollador junior, necesito transformar VS Code en un entorno de desarrollo profesional que garantice calidad, consistencia y productividad. El objetivo es instalar y configurar extensiones clave que automaticen el formateo y el análisis estático del código para cumplir con los estándares técnicos de la agencia.

Esta tarea incluye la instalación de herramientas de linting y formateo, así como la activación de funcionalidades automáticas como “format on save” y corrección automática de errores básicos.

Objetivo técnico:
Reducir errores humanos, mantener consistencia de estilo y mejorar la legibilidad del código desde el primer día.

Criterios de aceptación:

El código se formatea automáticamente al guardar.

Los errores de sintaxis y estilo aparecen en tiempo real.

La configuración queda persistida en settings.json.

🔹 Issue 1.2 – Configuración de reglas de linting

Título: Definición y configuración de reglas ESLint y Prettier

Descripción:
Para asegurar uniformidad en todo el equipo, es necesario definir reglas claras de estilo y calidad mediante archivos de configuración .eslintrc y .prettierrc. Esta tarea implica establecer reglas como uso obligatorio de punto y coma, tipo de comillas, indentación y detección de variables no utilizadas.

Objetivo técnico:
Estandarizar el código del proyecto y prevenir errores lógicos simples antes de la ejecución.

Criterios de aceptación:

El proyecto contiene archivos de configuración activos.

Las reglas generan advertencias o errores cuando no se cumplen.

No existen conflictos entre ESLint y Prettier.

🔹 Issue 1.3 – Reflexión técnica: Editor vs IDE

Título: Documentación técnica sobre entorno de desarrollo

Descripción:
Redactar un documento breve explicando la diferencia entre un editor de texto y un IDE, y justificar cómo VS Code puede comportarse como un entorno de desarrollo completo cuando se configura adecuadamente.

Objetivo técnico:
Desarrollar mentalidad profesional y comprensión conceptual de las herramientas utilizadas.

Criterios de aceptación:

Documento incluido en el README.md.

Explicación clara, técnica y argumentada.

🏁 MILESTONE 2 – Auditoría de Performance
🔹 Issue 2.1 – Análisis de rendimiento con Network

Título: Auditoría de recursos críticos en pestaña Network

Descripción:
Utilizar las DevTools del navegador para analizar la carga de recursos del sitio e-commerce. Se debe identificar archivos que impacten negativamente el tiempo de carga, como imágenes pesadas, scripts bloqueantes o respuestas lentas del servidor.

Se analizarán:

Tamaño de recursos

Tiempo de respuesta

Peticiones bloqueantes

Recursos sin compresión

Objetivo técnico:
Detectar cuellos de botella que afecten la experiencia del usuario y el SEO.

Criterios de aceptación:

Tabla con recursos problemáticos.

Identificación de al menos 3 mejoras potenciales.

Explicación técnica de cada hallazgo.

🔹 Issue 2.2 – Identificación de errores en Console

Título: Análisis y depuración de errores JavaScript

Descripción:
Revisar la pestaña Console en DevTools para detectar errores de ejecución que puedan estar afectando la experiencia del usuario. Esto incluye errores de tipo, recursos no encontrados (404), promesas no manejadas o variables indefinidas.

Para cada error detectado se deberá explicar:

Qué significa el error.

En qué parte del código se produce.

Cómo impacta al usuario.

Posible solución técnica.

Objetivo técnico:
Mejorar estabilidad y confiabilidad del sitio.

Criterios de aceptación:

Mínimo dos errores documentados.

Explicación clara del impacto técnico y funcional.

🔹 Issue 2.3 – Informe técnico de auditoría

Título: Elaboración de informe de diagnóstico técnico

Descripción:
Redactar un informe estructurado que consolide los hallazgos de la auditoría de rendimiento y errores detectados. El informe debe tener enfoque profesional y orientado a negocio, explicando cómo los problemas impactan la conversión y experiencia de usuario.

Estructura esperada:

Resumen ejecutivo

Problemas de rendimiento

Problemas de ejecución

Impacto en UX y negocio

Recomendaciones técnicas priorizadas

Objetivo técnico:
Traducir análisis técnico en decisiones estratégicas.

Criterios de aceptación:

Documento claro, estructurado y profesional.

Recomendaciones accionables.

🏁 MILESTONE 3 – Iniciación del Microservicio
🔹 Issue 3.1 – Inicialización del proyecto con npm y Git

Título: Creación y configuración inicial del microservicio

Descripción:
Crear la base estructural del nuevo microservicio utilizando npm para la gestión de dependencias y Git para el control de versiones. Esta tarea incluye la creación del directorio del proyecto, inicialización del package.json y repositorio Git.

Objetivo técnico:
Garantizar trazabilidad y gestión estructurada desde el inicio del proyecto.

Criterios de aceptación:

package.json generado correctamente.

Repositorio Git inicializado.

Primer commit realizado.

🔹 Issue 3.2 – Instalación y gestión de dependencias

Título: Configuración de dependencias de producción y desarrollo

Descripción:
Instalar las dependencias necesarias para el funcionamiento del microservicio (por ejemplo, Express) y herramientas de desarrollo como nodemon, ESLint y Prettier.

Se debe diferenciar correctamente:

Dependencias de producción.

Dependencias de desarrollo.

Objetivo técnico:
Mantener un entorno limpio y profesional optimizado para producción.

Criterios de aceptación:

package.json correctamente clasificado.

node_modules generado.

Dependencias visibles en archivo de configuración.

🔹 Issue 3.3 – Configuración de scripts npm

Título: Automatización de tareas mediante scripts npm

Descripción:
Configurar scripts en package.json para facilitar ejecución del proyecto en distintos entornos (desarrollo y producción) y permitir análisis de calidad mediante linting.

Objetivo técnico:
Optimizar flujo de trabajo y reducir comandos manuales repetitivos.

Criterios de aceptación:

npm run dev ejecuta el servidor con nodemon.

npm run start ejecuta entorno productivo.

npm run lint analiza el código.

🔹 Issue 3.4 – Estructura profesional del proyecto

Título: Definición de arquitectura base del microservicio

Descripción:
Crear una estructura modular y escalable separando responsabilidades en carpetas como controllers, routes y services. Esto facilitará el mantenimiento y crecimiento futuro del sistema.

Objetivo técnico:
Aplicar principios de arquitectura limpia desde el inicio.

Criterios de aceptación:

Estructura de carpetas implementada.

Separación clara de responsabilidades.

README con explicación básica de arquitectura.
